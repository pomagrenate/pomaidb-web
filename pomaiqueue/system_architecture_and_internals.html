System Architecture

System: Pomai Queue (Core Engine)
Component: Single-host Durable Queue
Status: Active Development

High-Level Design

PomaiQueue is a single-host durable queue with sharded actor threads. The core engine resides in src/core, while the gRPC server acts as a thin wrapper.

Producer → [Shard Hash] → [Shard Actor] → [Segment Log]

Consumer → [Shard Hash] → [Group State] → [Inflight] → [Checkpoint]

The shard actor currently executes tasks synchronously using a queue and worker thread. This architecture serves as a placeholder for more advanced scheduling mechanisms.

Durability & Persistence

The system guarantees durability through an append-only log structure validated by CRC64 checksums.

Implemented Mechanisms

  • Segment Log: Append-only storage with append/recover/read capabilities.
  • Checkpoint Store: Uses atomic rename and fsync to persist committed offsets.
  • Recovery: Scans segments, validates CRC, and truncates partial tails to ensure consistency.

Engine Contract & Replay

The engine operationalizes the queue contract through strict internal guarantees.

Core Guarantees

  • Deterministic Replay: State is reconstructed from the segment log plus an append-only transition log (<group>.events).
  • Bounded Scheduling: Uses a fixed-size deadline wheel to manage retries, avoiding expensive full inflight scans.
  • Durable Snapshots: <group>.state accelerates restart but does not define semantics; truth comes from the log.

Explain API

The InspectMessage(queue, group, id) function exposes internal state for debugging, including sequence, timestamps, retry counts, and lease ownership tokens.

Protocol Definition

The authoritative API contract is defined in src/rpc/proto/pomaiqueue.proto. Currently, proto definitions for queue lifecycle, produce/consume, and ack/nack are implemented.
